How did it start ?

During my first hackathon, the Starknet Hacker House, hosted by the Starknet Foundation, our initial idea was to build a ZKML Marketplace. However, we soon realized a significant challenge: if the ML model is on-chain, it becomes visible to everyone globally, allowing anyone to replicate and create their own model. To address this, we used Giza, as it supports Cairo, making it provable using STARK. Our exploration into Giza was aimed at understanding how we could leverage this technology to develop a ZKML Marketplace.

We were fortunate to have Danilo from Kakarot as our mentor for this event which he is a great mentor ü´°. On the first day, we managed to outline the marketplace's workflow, envisioning it more as an infrastructure service product. This setup would allow users to call for inference on a private model, receive the result along with a proof, and then verify it independently using STARK. The STARK prover we utilized was the Lambdaworks Platinum Prover. It was an exciting experience to experiment with this prover. We integrated it into an API and CLI, enabling users to generate a proof effortlessly. Once a proof is generated, the user can verify it independently to confirm if the proof is valid.


What is the problem we're solving ?

  • ZKMLs that are deployed on Starknet are ‚Äúpublic‚Äù and can be used by everyone

    What this means is that the user cant make the model private and take advantage to monetize the model so that people could use it.

  • Steep learning curve to get it running

    If they were to do it on their own it will be hard for them to run it on their own since so many things needs to be set up.

  • Most off-chain ML platforms are technically a blackbox

    There is no way to verify that they are using the actual model, e.g. GPT-4. No information on whether or not they changed one of the weights. it means that everything relies on trust and we wont know which version of the model you really using.


Example how its being used

lets say we have a game of Tick Tac Toe and every time you call an inference you wouldn‚Äôt want to generate a proof then return result only if the proof is done generated, but what we did was a user will be able to get the inference result and our service will generate the proof in parallel and once the game is done it will send back the user all the proof that is generated for every inference. But the second problem would be not everyone would want to proof on every inference being called. So what we had in mind was to give an option on when the user wanted to generate a proof. They could just call the inference and not generate a proof which is a win win where user don‚Äôt need to waste so much computing power to generate proof when the user doesn't need it. But when the user needs the proof it will then be generated and given back to the user to verify.


Reason why this is interesting ?

On-Chain Games and AW

This is where things gets interesting , imagine where people can have on-chain games or AW (autonomous world) where they can use a AI agent or ZKML on the games and just call it using our service and they can just pay by use anytime without setting up anything. We will support SDK and extension so that people could easily use it. This open up for more ideas to be build upon ZKML. This is why we gave an example on a Tic Tac Toe game for example because we saw a potential on using it on on-chain games and autonomous world.

Other use cases

But this doesn‚Äôt mean its just limited to just games but it can be use for others. For example we want to use ZKML to call inference to predict a user‚Äôs credit scoring. What i had in mind to always wanted to build for my first ever ZKML project was to build a on-chain credit scoring where it uses storage proof to query historical data on chain and feed it into a ZKML model to genrate the result of the user‚Äôs credit scoring and give back the proof to the user to verify that inference and its being computed correctly. So this is an example of not using for just games but it can be use for others too.
